Visión General del Sistema

El Sistema de Gestión Académica Avanzada es una aplicación fullstack que administra programas académicos, estudiantes, docentes, asignaturas, períodos, asignaciones, matrículas y calificaciones. Utiliza 3 servicios Docker orquestados con docker-compose.yml.

NestJS 10 Next.js 16 PostgreSQL 16 Docker Prisma 7.5

Diferencia clave con la versión anterior

Esta versión avanzada elimina la capa Repository. Los Services inyectan PrismaService directamente y ejecutan las consultas a la BD. Esto simplifica la arquitectura a 3 capas: Controller → Service (con Prisma) → PostgreSQL.

Docker y Orquestación

Todo el sistema se levanta con un solo comando: docker compose up -d. Docker Compose orquesta los 3 servicios en el orden correcto, usando healthcheck para garantizar que PostgreSQL esté listo antes de que el backend inicie.

¿Por qué healthcheck?

Sin healthcheck, el backend intentaría conectarse a PostgreSQL antes de que esté listo, causando errores de conexión. El healthcheck ejecuta pg_isready cada 3 segundos y solo permite que el backend inicie cuando PostgreSQL responde correctamente.

Base de Datos y Prisma ORM

La base de datos contiene 8 entidades conectadas mediante Foreign Keys. Prisma ORM genera el cliente TypeScript tipo-safe a partir del archivo schema.prisma.

Relaciones clave

  • ProgramaAcademico → Estudiante/Asignatura: Un programa tiene muchos estudiantes y asignaturas (1:N).
  • AsignacionDocente: Une Docente + Asignatura + Período con restricción @@unique compuesta.
  • Matrícula: Une Estudiante + AsignacionDocente. Un estudiante no se puede matricular dos veces en la misma asignación.
  • Calificación: Relación 1:1 con Matrícula (matriculaId @unique).

Backend — NestJS 10

El backend usa una arquitectura de 3 capas donde cada Service inyecta PrismaService directamente. Haz clic en cada paso del diagrama para ver la explicación detallada:

👤 Cliente (Navegador)

El usuario interactúa con el frontend (Next.js). Cuando hace clic en "Crear" o "Guardar", la función fetch() en lib/api.ts envía una petición HTTP (GET, POST, PUT, DELETE) con los datos en formato JSON al backend en el puerto 3001.

Capas del Backend — Código Real

Los DTOs (Data Transfer Objects) definen qué datos acepta cada endpoint. Usan decoradores de class-validator para validación automática.

create-programa-academico.dto.ts import { IsString, IsInt, IsNotEmpty, Min } from 'class-validator'; export class CreateProgramaAcademicoDto { @IsString() @IsNotEmpty() nombre: string; @IsString() @IsNotEmpty() codigo: string; @IsString() @IsNotEmpty() facultad: string; @IsInt() @Min(1) duracionSemestres: number; }

¿Qué hace cada decorador?

  • @IsString() → El campo debe ser un string. Si envían un número, retorna HTTP 400.
  • @IsNotEmpty() → No acepta strings vacíos "".
  • @IsInt() + @Min(1) → Debe ser entero mayor o igual a 1.

El Controller recibe peticiones HTTP y delega al Service. No contiene lógica de negocio.

programa-academico.controller.ts @Controller('programa-academico') export class ProgramaAcademicoController { constructor(private readonly service: ProgramaAcademicoService) {} @Post() create(@Body() dto: CreateProgramaAcademicoDto) { return this.service.create(dto); } @Get() findAll() { return this.service.findAll(); } @Get(':id') findOne(@Param('id', ParseIntPipe) id: number) { return this.service.findOne(id); } @Put(':id') update(@Param('id', ParseIntPipe) id: number, @Body() dto: UpdateProgramaAcademicoDto) { return this.service.update(id, dto); } @Delete(':id') remove(@Param('id', ParseIntPipe) id: number) { return this.service.remove(id); } }

Decoradores HTTP

  • @Controller('programa-academico') → Ruta base: /api/v1/programa-academico
  • @Post(), @Get(), @Put(':id'), @Delete(':id') → Verbos HTTP
  • @Body() → Extrae el body JSON de la petición
  • @Param('id', ParseIntPipe) → Extrae el parámetro de la URL y lo convierte a número

El Service contiene la lógica de negocio y usa PrismaService directamente (sin capa Repository intermedia).

programa-academico.service.ts @Injectable() export class ProgramaAcademicoService { constructor(private readonly prisma: PrismaService) {} async create(dto: CreateProgramaAcademicoDto) { // 1. Verificar que el código no esté duplicado const exists = await this.prisma.programaAcademico.findUnique({ where: { codigo: dto.codigo }, }); if (exists) { throw new ConflictException(`El código '${dto.codigo}' ya está en uso`); } // 2. Crear directamente con Prisma return this.prisma.programaAcademico.create({ data: dto }); } findAll() { return this.prisma.programaAcademico.findMany({ orderBy: { nombre: 'asc' }, }); } async findOne(id: number) { const programa = await this.prisma.programaAcademico.findUnique({ where: { id }, // include = LEFT JOIN → trae relaciones anidadas include: { estudiantes: true, asignaturas: true }, }); if (!programa) { throw new NotFoundException(`ProgramaAcademico #${id} no encontrado`); } return programa; } async update(id: number, dto: UpdateProgramaAcademicoDto) { await this.findOne(id); // Lanza 404 si no existe if (dto.codigo) { const existing = await this.prisma.programaAcademico.findFirst({ where: { codigo: dto.codigo, NOT: { id } }, }); if (existing) throw new ConflictException(`Código duplicado`); } return this.prisma.programaAcademico.update({ where: { id }, data: dto }); } async remove(id: number) { await this.findOne(id); return this.prisma.programaAcademico.delete({ where: { id } }); } }

Sin capa Repository

A diferencia de la versión anterior, aquí el Service llama a this.prisma.modelo.método() directamente. Esto simplifica el código pero acopla la lógica de negocio al ORM. Es el patrón recomendado por NestJS para proyectos medianos.

Los Módulos agrupan Controller + Service. PrismaModule es @Global(), por lo que no necesita importarse en cada módulo.

programa-academico.module.ts @Module({ controllers: [ProgramaAcademicoController], providers: [ProgramaAcademicoService], }) export class ProgramaAcademicoModule {}
prisma.module.ts @Global() // ← Disponible en TODA la app sin importar @Module({ providers: [PrismaService], exports: [PrismaService], }) export class PrismaModule {}

El archivo main.ts configura toda la aplicación: prefijo global, CORS, validación, manejo de errores y formato de respuestas.

main.ts async function bootstrap() { const app = await NestFactory.create(AppModule); // Prefijo global: todas las rutas empiezan con /api/v1 app.setGlobalPrefix('api/v1'); // Permite peticiones desde el frontend (puerto 3000) app.enableCors(); // Validación automática con class-validator app.useGlobalPipes(new ValidationPipe({ whitelist: true, // Elimina campos no definidos en DTO forbidNonWhitelisted: true, // Error si envían campos extra transform: true, // Convierte "1" a número 1 })); // Formato uniforme de errores app.useGlobalFilters(new HttpExceptionFilter()); // Formato uniforme de respuestas exitosas app.useGlobalInterceptors(new ResponseInterceptor()); await app.listen(3001); } bootstrap();

Caso Especial: Calificaciones

Fórmula de Nota Definitiva

Las calificaciones se calculan automáticamente en el Service usando la fórmula ponderada:

Definitiva = Nota1 × 0.30 + Nota2 × 0.30 + Nota3 × 0.40

calificacion.service.ts private calcularNotaDefinitiva(nota1: number, nota2: number, nota3: number): number { return Math.round((nota1 * 0.3 + nota2 * 0.3 + nota3 * 0.4) * 100) / 100; } async create(dto: CreateCalificacionDto) { // 1. Verificar que la matrícula existe const matricula = await this.prisma.matricula.findUnique({...}); if (!matricula) throw new NotFoundException(...); // 2. Verificar que no tenga calificación ya const exists = await this.prisma.calificacion.findUnique({...}); if (exists) throw new ConflictException('Ya existe calificación'); // 3. Calcular nota definitiva automáticamente const notaDefinitiva = this.calcularNotaDefinitiva(dto.nota1, dto.nota2, dto.nota3); return this.prisma.calificacion.create({ data: { ...dto, notaDefinitiva }, }); }

Reglas de Negocio Importantes

🔗
Asignación Docente — Triple FK

Valida que existan Docente, Asignatura y Período antes de crear. Usa Promise.all() para validar en paralelo y @@unique compuesto para evitar duplicados.

📅
Período Activo — Solo Uno

Al activar un período, el Service ejecuta updateMany({ activo: true } → { activo: false }) primero, garantizando que solo un período esté activo.

📝
Matrícula — No cancelar con notas

Al cancelar una matrícula, verifica si tiene calificación asociada. Si existe, lanza BadRequestException impidiendo la cancelación.

Frontend — Next.js 16 + React 19

El frontend utiliza App Router de Next.js con páginas "use client" que manejan estado con React hooks. Cada módulo tiene su propia página con formulario + tabla CRUD.

Código del Frontend

El cliente HTTP centralizado maneja todas las peticiones. Desenvuelve json.data automáticamente para que las páginas reciban los datos limpios.

lib/api.ts const BASE_URL = process.env.NEXT_PUBLIC_API_URL ?? "http://localhost:3001/api/v1"; async function request<T>(method: HttpMethod, path: string, body?: unknown): Promise<T> { const res = await fetch(`${BASE_URL}${path}`, { method, headers: { "Content-Type": "application/json" }, body: body !== undefined ? JSON.stringify(body) : undefined, }); if (!res.ok) { const error = await res.json().catch(() => ({})); throw new ApiError(res.status, error.message ?? ["Error desconocido"]); } const json: ApiResponse<T> = await res.json(); return json.data; // ← Desenvuelve automáticamente } export const api = { get: <T>(path: string) => request<T>("GET", path), post: <T>(path: string, body: unknown) => request<T>("POST", path, body), put: <T>(path: string, body: unknown) => request<T>("PUT", path, body), patch: <T>(path: string, body: unknown) => request<T>("PATCH", path, body), delete: <T>(path: string) => request<T>("DELETE", path), };

Cada Service del frontend encapsula las rutas de un módulo específico. Las páginas nunca llaman a api.get() directamente.

services/programas.service.ts import { api } from "../lib/api"; const BASE = "/programa-academico"; export const programasService = { findAll: () => api.get(BASE), findOne: (id: number) => api.get(`${BASE}/${id}`), create: (data: unknown) => api.post(BASE, data), update: (id: number, data: unknown) => api.put(`${BASE}/${id}`, data), remove: (id: number) => api.delete(`${BASE}/${id}`), };

Cada página usa "use client" + React hooks para manejar el estado del formulario, la lista de datos y las notificaciones.

app/estudiantes/page.tsx (patrón) "use client"; import { useState, useEffect } from "react"; import { estudiantesService, programasService } from "@/services"; export default function EstudiantesPage() { const [estudiantes, setEstudiantes] = useState([]); const [programas, setProgramas] = useState([]); const [loading, setLoading] = useState(true); const [showForm, setShowForm] = useState(false); const [editing, setEditing] = useState(null); const [form, setForm] = useState({...}); // Carga inicial: datos + catálogos en paralelo const load = async () => { const [est, prog] = await Promise.all([ estudiantesService.findAll(), programasService.findAll(), ]); setEstudiantes(est); setProgramas(prog); }; useEffect(() => { load(); }, []); const handleSubmit = async (e) => { e.preventDefault(); if (editing) { await estudiantesService.update(editing.id, form); } else { await estudiantesService.create(form); } notify("Operación exitosa"); // Desaparece en 3s resetForm(); load(); // Recarga tabla }; return ( <div> {/* Botón nuevo + Formulario condicional + Tabla */} </div> ); }

Patrón dual Create/Edit

Todas las páginas usan un solo formulario para crear y editar. Si editing es null, llama a service.create(). Si tiene un objeto, llama a service.update(). El botón "Editar" en la tabla llena el formulario con setForm(datos) y setEditing(item).

Flujo Completo del Negocio

¿Cómo se encadenan las operaciones en el sistema? Siguiendo el flujo académico real:

Orden de dependencia

No se puede crear un Estudiante sin que exista un Programa. No se puede matricular sin que exista la Asignación del docente. Las Foreign Keys en la base de datos garantizan la integridad referencial.

Glosario de Conceptos

Haz clic en cada concepto para expandir la explicación:

Inyección de Dependencias (DI)

En lugar de que una clase cree sus propias dependencias (new ProgramaAcademicoService()), NestJS las inyecta automáticamente en el constructor. Esto funciona gracias al decorador @Injectable() y al registro en providers del módulo.

// NestJS hace esto automáticamente: const prisma = new PrismaService(); const service = new ProgramaAcademicoService(prisma); const controller = new ProgramaAcademicoController(service);
ValidationPipe + class-validator

El ValidationPipe (configurado globalmente en main.ts) intercepta el body de cada petición y lo valida contra el DTO usando decoradores de class-validator. Si falla, retorna HTTP 400 sin ejecutar ninguna lógica.

  • whitelist: true → Elimina campos que no estén en el DTO
  • forbidNonWhitelisted: true → Error si envían campos extra
  • transform: true → Convierte "1" a 1
HttpExceptionFilter

Captura todas las excepciones HTTP y las formatea en un JSON uniforme. Sin este filtro, cada tipo de error tendría un formato diferente.

// Formato uniforme de error: { "statusCode": 404, "message": "Estudiante con ID 99 no encontrado", "error": "NotFoundException", "path": "/api/v1/estudiante/99", "timestamp": "2026-04-12T10:30:00.000Z" }
ResponseInterceptor

Envuelve todas las respuestas exitosas en la estructura { statusCode, message, data }. El frontend siempre sabe que el dato real está en json.data.

'use client' en Next.js

Next.js renderiza componentes en el servidor por defecto. Cuando un componente necesita useState, useEffect o eventos del navegador (onClick), debe marcarse con 'use client' al inicio del archivo para indicar que se ejecuta en el navegador.

Prisma — include y @relation

@relation en el schema define las Foreign Keys. include: { programaAcademico: true } en el código equivale a un LEFT JOIN SQL, trayendo la entidad relacionada anidada en el resultado JSON.

// Prisma findUnique con include: { id: 1, nombres: "Juan", programaAcademico: { id: 1, nombre: "Ing. Sistemas" } } // Equivale a SQL: SELECT e.*, p.* FROM estudiantes e LEFT JOIN programas_academicos p ON e.programaAcademicoId = p.id
CORS — Cross-Origin Resource Sharing

El frontend (puerto 3000) y el backend (puerto 3001) son orígenes diferentes. Sin CORS habilitado, el navegador bloquearía las peticiones. En main.ts, app.enableCors() autoriza al frontend a comunicarse con el backend.

Errores de Prisma — P2002 y P2003
  • P2002 → Violación de UNIQUE. Ejemplo: intentar crear dos estudiantes con el mismo codigoEstudiantil. El Service verifica antes de crear y lanza ConflictException (HTTP 409).
  • P2003 → Violación de FOREIGN KEY. Ejemplo: intentar eliminar un programa que tiene estudiantes asociados.
@Global() en NestJS

El decorador @Global() en PrismaModule hace que PrismaService esté disponible en todos los módulos sin necesidad de importarlo explícitamente. Por eso los Services pueden inyectar PrismaService directamente en su constructor.
