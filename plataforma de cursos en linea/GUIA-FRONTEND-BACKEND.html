La Gran Analogía: El Restaurante 🍕

🧒 Imagínalo así...

Piensa que el sistema es un restaurante. Tú eres el cliente, el Frontend es el mesero que te toma el pedido, el Backend es el cocinero que prepara la comida, y la Base de Datos es la nevera donde están todos los ingredientes.

🖥️
Frontend — El Mesero

Es lo que ves y tocas. Muestra la pantalla, los botones, los formularios. Cuando haces clic en "Guardar estudiante", el mesero (Next.js) toma tu pedido y se lo lleva a la cocina.

Next.js + React Puerto 3000
👨‍🍳
Backend — El Cocinero

Es la cocina, no la ves. Recibe el pedido del mesero, lo procesa, revisa si ya existe en la nevera, y devuelve el resultado. Trabaja con reglas: "no puede haber dos estudiantes con el mismo código".

NestJS Puerto 3001
🗄️
Base de Datos — La Nevera

Guarda todo de forma permanente. Cuando apagas el computador, los datos siguen ahí. El cocinero (Backend) es el único que abre la nevera — el mesero nunca la toca directamente.

PostgreSQL Prisma ORM
📋
La Comanda — HTTP/JSON

El mesero le pasa la comanda al cocinero en un formato especial llamado JSON: {"nombres": "Ana", "apellidos": "López"}. La ruta HTTP dice qué quiere el cliente: POST /api/v1/estudiante = "quiero crear uno nuevo".

useState y useEffect — La Memoria del Mesero

🧒 Para entenderlo fácil...

useState es la libreta del mesero donde anota lo que está pasando ahora mismo (la lista de estudiantes, si hay error, si está cargando). useEffect es la acción que el mesero hace automáticamente cuando llega al turno: "apenas abra la página, voy a ir a la cocina a traer todos los estudiantes".

📓 useState — La Libreta de Notas

📝
useState([]) — La lista

Guarda la lista de estudiantes. Al principio está vacía []. Cuando llegan los datos del backend, React actualiza la pantalla automáticamente.

const [estudiantes, setEstudiantes] = useState(Estudiante[]>([]);
⏳
useState(true) — ¿Cargando?

Empieza en true (cargando). Cuando los datos llegan, se pone en false y desaparece el esqueleto de carga. El usuario ve la tabla real.

const [loading, setLoading] = useState(boolean>(true);
❌
useState(null) — ¿Hay error?

Empieza en null (sin error). Si el backend devuelve un fallo, guarda el mensaje aquí y React lo muestra en pantalla en rojo.

const [error, setError] = useState<string | null>(null);
✅
useState(null) — ¡Éxito!

Cuando el backend confirma que guardó el estudiante, aquí se guarda el mensaje de éxito que sale en verde por 3 segundos y luego desaparece solo.

const [success, setSuccess] = useState<string | null>(null);

⚡ useEffect — El Turno Automático

¿Qué hace exactamente useEffect?

Ejecuta código después de que la página se pinta en pantalla. El arreglo vacío [] al final significa "solo hazlo una vez, cuando la página cargue por primera vez". Sin ese arreglo, se ejecutaría infinitamente.

frontend/src/app/estudiantes/page.tsx
// ① La función "load" va a buscar los datos al backend const load = async () => { try { // Pide estudiantes Y programas al mismo tiempo (más rápido) const [est, prog] = await Promise.all([ estudiantesService.findAll(), // GET /api/v1/estudiante programasService.findAll(), // GET /api/v1/programa-academico ]); setEstudiantes(est); // ② Actualiza la libreta: lista de estudiantes setProgramas(prog); // ③ Actualiza la libreta: lista de programas } catch { setError("Error al cargar los datos"); // ④ Si falla, guarda el error } finally { setLoading(false); // ⑤ Siempre quita el "cargando..." } }; // ⑥ useEffect le dice a React: "llama a load() apenas pinte la página" useEffect(() => { load(); }, []); // ↑ [] = solo la primera vez

El Servicio y lib/api.ts — El Teléfono del Mesero

🧒 Para entenderlo fácil...

El mesero no camina hasta la cocina cada vez que quiere pedir algo. Tiene un teléfono especial con botones pre-configurados: un botón que dice "traer todos los estudiantes", otro que dice "crear uno nuevo". Ese teléfono es lib/api.ts. Los botones son estudiantesService.

estudiantesService — Los Botones del Teléfono
// frontend/src/services/estudiantes.service.ts import { api } from "@/lib/api"; export const estudiantesService = { // 📋 Traer todos → GET /api/v1/estudiante findAll: () => api.get<Estudiante[]>("/estudiante"), // 👤 Traer uno → GET /api/v1/estudiante/5 findOne: (id: number) => api.get<Estudiante>(`/estudiante/${id}`), // ➕ Crear → POST /api/v1/estudiante create: (data) => api.post<Estudiante>("/estudiante", data), // ✏️ Editar → PUT /api/v1/estudiante/5 update: (id, data) => api.put<Estudiante>(`/estudiante/${id}`, data), // 🗑️ Borrar → DELETE /api/v1/estudiante/5 remove: (id) => api.delete<void>(`/estudiante/${id}`), };
lib/api.ts — El Teléfono en Sí
// frontend/src/lib/api.ts const BASE_URL = process.env .NEXT_PUBLIC_API_URL ?? "http://localhost:3001/api/v1"; // 🔧 Función que hace la llamada HTTP real async function request<T>(method, path, body?) { const res = await fetch(`${BASE_URL}${path}`, { method, headers: { "Content-Type": "application/json" }, body: body ? JSON.stringify(body) : undefined, }); if (!res.ok) { // ❌ Cocina dijo que no → lanza error throw new ApiError(res.status, [...]); } // ✅ Extrae solo "data" de la respuesta const json = await res.json(); return json.data; // { statusCode, message, data } } export const api = { get: (path) => request("GET", path), post: (path, body) => request("POST", path, body), put: (path, body) => request("PUT", path, body), delete: (path) => request("DELETE", path), };

¿Por qué separar servicio y api.ts?

Si mañana el backend cambia de puerto, solo cambias una línea en lib/api.ts y todo el resto funciona igual. Si quieres agregar un módulo nuevo, solo creas un nuevo servicio de 5 líneas. Un solo lugar de control = menos errores.

El Módulo Backend — La Cocina por Dentro

🧒 Para entenderlo fácil...

La cocina tiene 4 personas con roles distintos: el portero (Controller) que recibe el pedido y lo valida, el chef (Service) que decide cómo prepararlo con las reglas del restaurante, el despensero (Prisma) que habla con la nevera, y el gerente (Module) que organizó a los tres y los presentó entre sí.

EstudianteModule — El Gerente

El Module es quien presenta a todos entre sí. Le dice a NestJS: "este Controller y este Service trabajan juntos". Sin el Module, NestJS no sabría que existen.

estudiante.module.ts
// El módulo "registra" al controller y al service @Module({ controllers: [EstudianteController], // 🚪 "este es el portero" providers: [EstudianteService], // 👨‍🍳 "este es el chef" }) export class EstudianteModule {} // PrismaService NO necesita estar aquí porque es @Global() // Está disponible en toda la aplicación automáticamente

EstudianteController — El Portero

El Controller solo recibe la solicitud HTTP y la pasa al Service. No tiene lógica de negocio — si le preguntas "¿puede crearse este estudiante?", no sabe. Eso lo decide el Service.

estudiante.controller.ts
@Controller('estudiante') // → todas las rutas empiezan con /api/v1/estudiante export class EstudianteController { constructor(private readonly service: EstudianteService) {} // ↑ NestJS inyecta el Service automáticamente (Dependency Injection) @Get() // GET /api/v1/estudiante findAll() { return this.service.findAll(); // delega al chef } @Post() // POST /api/v1/estudiante create(@Body() dto: CreateEstudianteDto) { // @Body() extrae el JSON del cuerpo de la petición return this.service.create(dto); } @Put(':id') // PUT /api/v1/estudiante/5 update(@Param('id', ParseIntPipe) id: number, @Body() dto) { // ParseIntPipe convierte "5" (texto) a 5 (número) return this.service.update(id, dto); } @Delete(':id') // DELETE /api/v1/estudiante/5 remove(@Param('id', ParseIntPipe) id: number) { return this.service.remove(id); } }

EstudianteService — El Chef con las Reglas

Aquí vive la lógica real. Antes de crear un estudiante, el Service verifica que no exista duplicado y que el programa académico exista. Si algo falla, lanza una excepción y el backend responde con error 409 o 404.

estudiante.service.ts (fragmento create)
@Injectable() export class EstudianteService { constructor(private readonly prisma: PrismaService) {} // ↑ PrismaService inyectado: el "despensero" async create(dto: CreateEstudianteDto) { // 1️⃣ ¿Ya existe alguien con el mismo código/documento/correo? const existing = await this.prisma.estudiante.findFirst({ where: { OR: [ { codigoEstudiantil: dto.codigoEstudiantil }, { documentoIdentidad: dto.documentoIdentidad }, ]} }); if (existing) throw new ConflictException("Ya existe..."); // ↑ Esto devuelve HTTP 409 al frontend automáticamente // 2️⃣ ¿Existe el programa académico? const programa = await this.prisma.programaAcademico.findUnique({ where: { id: dto.programaAcademicoId } }); if (!programa) throw new NotFoundException("Programa no encontrado"); // ↑ Esto devuelve HTTP 404 al frontend // 3️⃣ Todo ok → crear en la base de datos return this.prisma.estudiante.create({ data: { ...dto, fechaNacimiento: new Date(dto.fechaNacimiento) }, include: { programaAcademico: { select: { id: true, nombre: true } } } }); } }

DTO — El Formulario de Pedido

DTO significa Data Transfer Object. Es como el formulario impreso que el mesero le da al cliente para llenar: define exactamente qué campos son obligatorios y en qué formato deben venir. class-validator los verifica automáticamente.

dto/create-estudiante.dto.ts
import { IsString, IsEmail, IsInt, IsDateString, IsNotEmpty } from 'class-validator'; export class CreateEstudianteDto { @IsNotEmpty() @IsString() nombres: string; // ← obligatorio, debe ser texto @IsNotEmpty() @IsString() apellidos: string; @IsEmail() correoInstitucional: string; // ← debe ser un email válido @IsInt() programaAcademicoId: number; // ← debe ser un número entero @IsDateString() fechaNacimiento: string; // ← formato "2000-05-15" } // Si alguno falla → HTTP 400 Bad Request automático // El Service nunca recibe datos inválidos

Flujo Completo — Crear un Estudiante

🧒 El cuento completo paso a paso...

Ana quiere agregar a "Juan Pérez" al sistema. Llena el formulario y hace clic en "Guardar". Aquí está todo lo que pasa en menos de 1 segundo:

1

Usuario hace clic en "Guardar"

React llama a handleSubmit(). Esta función llama a estudiantesService.create(form). El form tiene los datos del formulario: nombres, apellidos, código, etc.

// en page.tsx const handleSubmit = async (e) => { e.preventDefault(); await estudiantesService.create(form); // ← paso 2 };
2

estudiantesService llama a lib/api.ts

El servicio llama a api.post("/estudiante", data). El cliente HTTP arma la petición: método POST, URL completa http://localhost:3001/api/v1/estudiante, cuerpo en JSON.

// fetch() sale del navegador hacia el backend POST http://localhost:3001/api/v1/estudiante Content-Type: application/json { "nombres": "Juan", "apellidos": "Pérez", "codigoEstudiantil": "20231001", "programaAcademicoId": 2 }
3

NestJS: ValidationPipe verifica el DTO

Antes de que llegue al Controller, el ValidationPipe global verifica que todos los campos del DTO sean válidos. Si falta alguno o tiene formato incorrecto, devuelve HTTP 400 de inmediato.

// main.ts — configurado una vez para todos los módulos app.useGlobalPipes(new ValidationPipe({ whitelist: true }));
4

Controller recibe la petición

El @Post() de EstudianteController captura la ruta. Extrae el body con @Body() y llama a service.create(dto). Nada más.

5

Service aplica las reglas de negocio

Verifica con Prisma si ya existe un estudiante con el mismo código o correo. Si existe → ConflictException (409). Verifica que el programa académico exista. Si no → NotFoundException (404).

6

Prisma ejecuta el INSERT en PostgreSQL

this.prisma.estudiante.create({ data: dto }) genera y ejecuta el SQL: INSERT INTO "Estudiante" (...) VALUES (...). PostgreSQL guarda el registro y devuelve el nuevo estudiante con su id autoincremental.

7

ResponseInterceptor envuelve la respuesta

El interceptor global toma el objeto devuelto por el Service y lo envuelve en el formato estándar:

{ "statusCode": 201, "message": "Created", "data": { "id": 5, "nombres": "Juan", "apellidos": "Pérez", "programaAcademico": { "id": 2, "nombre": "Ingeniería de Sistemas" } } }
8

Frontend recibe la respuesta y actualiza la pantalla

lib/api.ts extrae el campo data del JSON. handleSubmit llama a notify("Estudiante creado correctamente") → aparece el mensaje verde. Luego llama a load() para refrescar la tabla con los datos más recientes.

// en page.tsx — tras el create exitoso notify("Estudiante creado correctamente"); // setSuccess(msg) → 3s → null setShowForm(false); // cierra el formulario await load(); // refresca la tabla

Resumen en una frase

useState guarda lo que la página sabe ahora mismo. useEffect le dice "apenas aparezcas, ve a buscar los datos". El servicio tiene los botones del teléfono. lib/api.ts es el teléfono. El Controller recibe el pedido. El Service lo prepara con reglas. Prisma habla con la base de datos. Y todo vuelve como JSON al frontend.
